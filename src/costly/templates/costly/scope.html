Project Scope - City of Raleigh Municipal Site GSI/LID Cost Tool

Task 1 - Develop Cost Tool Architectural Concepts

Tetra Tech will use the Cost-Benefit Tool Concept document (Tetra Tech, 2016) developed with the City of Raleigh and background research as a foundation for this project. The project team will continue to research key aspects of existing tools (e.g., cost equations, input and output, friendly formats) as appropriate to incorporate into this new web-based application, and a preliminary architecture will be outlined to test against the research objective. It is expected that there will be multiple elements that the team gathers and evaluates during initial planning.

During the early portion of Task 1, Tetra Tech will work with the City to assemble a Project Advisory Committee (PAC). The purpose of the PAC will be to provide input and tool testing from intended tool users to enhance tool design and performance in achieving the City's objectives for the tool. Agreed upon architectural concepts will be documented to guide tool development and testing. Potential PAC members include City staff that oversee site plan designs for City property and private development (e.g., Construction Management, Development Services, Stormwater Management) along with third party designers associated with site development.

To identify and evaluate tool architectural concepts, the team will solicit input from the PAC to establish optimal inputs and outputs for the tool. The goal is to test the applicability of the cost tool with variations of inputs and outputs that will increase the likelihood of its use to assist municipalities and the development community during pre-development, site design, and design stages of review. The challenge of creating an accurate, user-friendly cost estimation tool is making sure that the tool remains simple while containing just enough detail to satisfy the intended user by showing that the application is a credible resource. Expectations for level of detail and accuracy at the three stages will be explored with the PAC.

Additionally, Tetra Tech will work with the PAC to conceptualize how the tool can best incorporate estimation of potential cost savings of smart design (e.g., LID) features such as narrowing street widths, eliminating curb and gutter, and reducing or eliminating drainage piping where swales can meet requirements. The goal is to keep the output of the tool simple so it translates into a resource that an average user desires to implement to make site assessments, but also evaluate traditional infrastructure, mixed, and GSI/LID projects fiscally to drive decision-making and negotiations.

Once the project team, with PAC input, decides upon project components to be represented in project cost estimates, the team will create a local cost database. The database will incorporate ideas stemming from a facilitated dialog between the project team and the PAC. Additionally, the research team will work with the PAC to decide how this database will be utilized in the tool and accessed by users. For example, the research team will need to figure out the accessibility of the data for different user types.

Based on the PAC input throughout Task 1, Tetra Tech will formulate and document the preliminary architecture concepts that will be used for testing and troubleshooting the tool through Task 2.

Task 1 Deliverable

  1. Document summarizing the architectural concepts to guide Task 2 tool construction

Task 2 - Construct the Prototype Cost Tool

Tetra Tech will build prototype techniques and tools that elaborate on the preliminary architecture concepts for select users to test before building the full working software. Once the requirements have been finalized and approved through the prototyping process, development of the application will begin. Based on the preliminary requirements, the application will be developed primarily in HTML and will be compatible with the major browsers and their two most current versions. The development will take into account browser dependencies, where possible, as well as interaction with plug-ins.

The project team will follow the agile development lifecycle (i.e., an industry standard comprising a set of values and principles for software development under which requirements and solutions evolve through the collaborative effort of self-organizing cross-functional teams) and the finalized requirements will be scoped into three incremental (6 week to 2 month) software releases. Each release will be fully tested internally before being provided to the City (PAC) for their release testing. Testing feedback will be collected, reviewed, and implemented based on priorities and funding. This iterative approach to development and testing allows the development team to rapidly receive and address feedback from the tool testers.

Key features that Tetra Tech will create and get feedback from the PAC include:

  1. The design of the web link for the user login. The initial thought is that the application will not be publicly accessible. It is proposed that user access be controlled through user authentication so all users will need a User Name and Password managed by the City of Raleigh.
  2. The design and content of the main home page with a brief description of the tool, purpose and how to use it, and contact information for main point of contact if the user needs help.
  3. The types of scenario inputs that the user would have the option to select and customize to describe the site for which the tool is being applied - For example, these may include:
    • SCM sizing information
    • Basic infrastructure quantities (e.g., paved road, curb and gutter, parking area, drainage piping)
    • Option to input user-defined unit cost information
  4. Formulating a basic results report that talks about:
    • Scenario description
    • BMP and infrastructure assumptions
    • Unit cost assumptions
    • Up front capital costs, O&M costs, and life-cycle costs for each element
    • Total costs
  5. Saving the scenario to the username with a sorting filter option.
  6. Allowing the user to choose multiple scenarios to compare costs (i.e., up to three scenarios for comparison).
  7. Enhanced viewing of results for a single scenario or multiple scenario comparisons.
  8. Enhanced administrative functions as resources allow. For example, add options that the City would like for administration of the user data.
  9. Input regarding the look and design of the tool (to the extent resources allow)

Task 2 Deliverables

  1. Documentation of web-based tool requirements

  2. Prototype tool (3 iterative versions) releases for testing

  3. Final project tool for publishing

Task 3 - Deliver the Municipal Site GSI/LID Cost Tool for Installation on City Server

Once development of the City of Raleigh Site GSI/LID Cost Tool is complete and all functionality has been tested and verified by the beta-testers, the team will follow basic software construction to help deploy the system into the City's production environment. The team has already engaged the City of Raleigh's IT Infrastructure staff to facilitate a streamlined and efficient transition from Tetra Tech's development environment to the City of Raleigh's production environment. When the application is ready to be moved to the Raleigh Production environment, Tetra Tech will work directly with the City's IT Infrastructure staff to support the deployment of the application. In support of this effort the application will be developed from the start in a manner that is compatible with the City's technical infrastructure requirements, security protocols, and user management platform. Documentation including source code will be provided. A User Manual, specified in the WRF "Web-based Product Criteria" is not anticipated as the intent is for the tool to be easy to use with any simple user guidance provided online.

Task 3 Deliverable

  1. Programmed tool for installation on City of Raleigh website along with installation guide
  2. Documentation of source code

Task 4 - Deliver Design Report

The primary deliverable for this research project will be the City of Raleigh Site GSI/LID Cost Tool web-based application. However, a Design Report will be provided for a project record. The report will briefly summarize the project background, objective, approach, and outcomes incorporating the architectural concepts, tool requirements, cost equations, and source code from the previous tasks. As appropriate, recommendations for how the initial product might be enhanced or complemented in the future will also be documented.

Task 4 Deliverable

  1. Design Report draft
  2. Design Report final
